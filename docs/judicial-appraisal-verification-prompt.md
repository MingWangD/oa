# 司法鉴定系统手册级功能验证提示词 (Prompt)

## 🎯 验证目标
严格按照《司法鉴定系统使用手册》以及 `docs/judicial-appraisal-flowchart-baseline.md` 登记的主流程图、细化流程图，对 19 个司法鉴定流程进行穷尽式功能验证，确保系统表现与手册业务逻辑、流程图流转路径完全一致，实现手册级精准验收。

## 🛠 核心验证要求

### 1. 19 个流程逐条真实跑通
- **深度穿透**：每一条流程必须完整历经手册规定的所有节点、角色、退回路径、分支判断、附件处理及归档环节。
- **流程图对照**：每次补充运行时测试前，必须先查看 `docs/judicial-appraisal-flowchart-baseline.md` 和 `docs/flowcharts/`，将图中的节点、判断、退回线、输出文件映射为测试场景。
- **杜绝假阳性**：严禁仅查看流程配置存在或仅跑通 happy path 主链 E2E；必须模拟真实业务操作进行全链路流转。

### 2. 条件分支全覆盖
必须构造特定数据以触发并验证以下所有决策逻辑：
- **受理决策**：是否受理
- **勘验决策**：是否初步勘验
- **鉴定决策**：是否具备鉴定条件
- **财务决策**：是否缴费、是否退费、是否 F 类项目
- **产物决策**：是否出具征求意见稿、是否收到异议函、是否邮寄、用章/审核是否通过等
- **流程关联**：补充材料、关联原流程处理等

### 3. 通用办理页功能闭环
- **基础操作**：保存、转交、退回、返回功能的逻辑正确性。
- **业务约束**：必填项校验（后端强制）、附件去重（MD5/内容识别）。
- **信息展示**：会签意见聚合展示、实时流程图、详细流程操作日志。
- **关联与权限**：父子流程关系展示、主办人/经办人权限隔离（经办人只读等）。

### 4. 附件与知识库归档完整性
- **归档深度**：表单快照、附件版本、审批意见、操作日志、父子流程关系必须按案件号自动归档。
- **质量要求**：不得出现漏归档、串案号、重复文件无法识别等问题。
- **可访问性**：节点附件必须在归档后可精准查询、预览及下载。

### 5. 查询、报表与导出一致性
- **功能模块**：我的工作、工作台统计、工作查询。
- **权限与范围**：严格执行行级数据权限（本人/本部门/全部等）。
- **数据一致性**：报表数据必须来源于真实办理结果；导出结果必须与页面查询条件、字段含义及手册模板保持高度一致。

### 6. 原 OA 测试数据驱动验证
- **真实样板**：至少抽取原 OA 系统中的 3-5 个真实案件样例，对比其字段含义、操作路径和最终流程结果。
- **等价性原则**：虽然 UI 表现可能不同，但业务路径、数据流向和办理结果必须与原系统及手册描述达成 100% 等价。

## 📝 验收标准
- **手册对齐度**：100%
- **分支覆盖率**：100%
- **数据准确性**：无错归档、无混淆数据
- **操作顺畅度**：符合手册描述的操作路径
