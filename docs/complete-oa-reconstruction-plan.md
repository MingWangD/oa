# 完整 OA 系统重构分阶段实施计划

更新时间：2026-06-10

## 1. 需求依据

冲突时按以下优先级执行：

1. 《完整OA系统重构需求规格说明书.docx》
2. 《司法鉴定系统使用手册.docx》及细化流程图
3. 现有 OA 网站的实际功能和行为
4. 当前项目代码
5. `docs/` 下旧 Markdown 文档

旧 Markdown 文档中的“司法鉴定一期简化方案”不再作为范围边界。

## 2. 当前现状与主要缺口

### 已有能力

- Spring Boot 3、Java 17、Spring Security、JWT、MyBatis-Plus 基础工程。
- Vue 3、TypeScript、Vite、Pinia、Vue Router、Element Plus 基础工程。
- 用户、部门、岗位、角色基础模型。
- 案件、流程实例、节点实例、任务、候选任务和基础状态机。
- 新建工作、我的工作、工作查询、知识库等页面骨架。
- 文件、台账、流程日志、审计相关数据库设计。

### 主要缺口

- 当前系统仍以司法鉴定一期案件系统为主，完整 OA 模块尚未实现。
- 权限仅有基础管理员判断，缺菜单/按钮/API 权限、数据权限和流程节点权限。
- 工作流仍以硬编码状态机为主，缺动态表单、动态流程、版本发布、并行/汇聚、子流程和完整退回规则。
- 文件、知识库、案件自动归档、版本、预览下载与权限尚未形成完整闭环。
- 司法鉴定 20 个流程和 19 个表单尚未逐节点落地。
- 数据报表、数据源、流程监控、委托、销毁、超时统计等平台能力缺失。
- CRM、合同、项目、仓库、行政、人资、公文、档案等完整 OA 业务模块缺失。

## 3. 分阶段计划

### 第一阶段：公共平台与可运行骨架

目标：让项目稳定构建，并建立承载完整 OA 范围的统一入口。

- 修复登录、组织、用户管理、候选任务等现有断裂引用。
- 建立完整 OA 菜单与模块范围台账。
- 建立司法鉴定 20 个流程、19 个表单和专属角色元数据目录。
- 补齐登录、首页、个人资料、用户管理、案件详情和完整 OA 菜单入口。
- 固化需求优先级和协作习惯。

验收：

- 后端 `mvn test` 通过。
- 前端 `npm run build` 通过。
- 管理员可看到完整 OA 菜单范围。
- 平台 API 可返回阶段计划、模块目录和司法鉴定流程/表单目录。

### 第二阶段：权限、动态表单与动态流程平台

目标：将硬编码状态机升级为可配置、可版本化的流程平台。

- RBAC 菜单、页面、按钮和接口权限。
- 本人、本部门、本部门及下级、自定义组织、全量数据权限。
- 动态表单字段、布局、校验、权限、联动、计算、附件、子表和版本。
- 动态流程开始/办理/审批/并行/汇聚/子流程/结束节点。
- 办理人规则、条件表达式、超时、退回、撤回、委托、认领和加签。
- 流程定义发布版本不可变，运行实例绑定版本。

验收：

- 无权限访问由后端拒绝。
- 管理员可创建、预览、发布、恢复表单与流程版本。
- 新流程版本不影响历史实例。

### 第三阶段：文件、知识与审计平台

目标：形成统一文件服务和案件知识归档闭环。

- MinIO 上传、下载、在线预览、查重、版本、水印和病毒扫描扩展。
- 个人、部门、公共知识目录及目录/文件权限。
- 每个流程节点完成后，表单与附件按案件号自动归档。
- 文件下载、预览、变更和权限操作全程审计。
- 审计日志不可由普通管理员删除或篡改。

验收：

- 无权限目录和文件不可见、不可下载。
- 流程重开时历史归档版本完整保留。
- 案件节点结果可在知识库按案件号检索。

### 第四阶段：司法鉴定完整业务高保真

目标：逐一落地 20 个流程和 19 个表单。

- 配置每个流程的输入文件、输出文件、注意事项和归档要求。
- 落实逐节点角色、分支条件、退回目标、并行任务、子流程和流程关联。
- 落实异议、出庭、退费、撤案、终止、暂停、用章和归档。
- 完整保留鉴定意见书初稿、A、A-B、A-B-C、最终送审稿。
- 落实项目金额、中心格式、F 类项目等条件规则。

验收：

- 20 个流程定义和 19 个表单定义均可发布和运行。
- 细化流程图中的逐级审核不得合并。
- 每个节点产物、意见、角色、时间和退回轨迹可追溯。

### 第五阶段：完整 OA 业务域

目标：按模块逐步复刻管理员可见的完整 OA。

- 个人事务、应用中心。
- CRM、绩效、合同、项目、仓库、安全风险。
- 行政办公、督查督办、智能门户、报表中心。
- 人力资源、考勤、公文、档案、交流园地。
- 开放与集成平台、附件程序、系统管理。

验收：

- 每个模块按需求规格书的主要页面、关键动作、状态权限和验收口径验收。
- 所有模块复用统一权限、附件、流程、消息、报表和审计能力。

### 第六阶段：迁移、集成、性能与上线治理

目标：完成生产可用和验收闭环。

- 历史数据和附件迁移工具、字段映射、校验报告和失败重试。
- 外部系统、SSO、统一待办和移动开放平台。
- 安全、权限、审计、性能、备份恢复和灾备演练。
- 部署、监控、告警和运维文档。

验收：

- 迁移结果可校验、可重试、可追溯。
- 关键接口满足性能与安全要求。
- 上线、回滚、备份和恢复流程经过演练。

## 4. 实施原则

- 公共平台能力优先于重复业务实现。
- 保留并扩展现有代码，不无理由推翻。
- 每次修改后运行可用的前后端测试和构建。
- 所有权限必须由后端强制执行。
- 所有关键业务动作、配置变更、审批、下载和权限变更必须审计。
