# 提示词 2：主干流程试跑

```text
请执行司法鉴定管理系统的“第一条主干流程手册级试跑”。

目标：
先打穿一条核心主干业务闭环，验证系统底座是否真正可运行。不要一开始铺开全部 19 个流程。先用真实业务账号、真实页面、真实流程实例跑通主干链路，发现手册范围内的问题必须当场修复并复验。

验收依据：
1. docs/司法鉴定系统使用手册.docx
2. docs/manual-scope-acceptance-template.md
3. docs/judicial-appraisal-master-checklist.md
4. docs/judicial-appraisal-flow-verification-matrix.md
5. docs/judicial-appraisal-flowchart-baseline.md
6. docs/browser-validation-record-2026-06-13.md
7. docs/original-oa-samples/
8. checklist.md
9. GEMINI.md
10. AGENT.md

测试数据要求：
项目中之前已经导出过 Excel 表格数据，可以用于测试。请优先使用 docs/original-oa-samples/exports/ 下与主干流程相关的 Excel 导出文件作为测试来源，并结合 docs/original-oa-samples/csv-preview/、docs/original-oa-samples/json/、docs/original-oa-samples/metadata/ 和 docs/original-oa-samples/raw/ 对照字段、流程名称、状态、查询结果和原 OA 操作习惯。不要优先凭空造数据；只有导出数据不足以完成主干闭环时，才补充最小必要测试数据，并说明原因。

主干流程建议：
收到委托书 → 收案/审阅/决定受理 → 分派/办理 → 编制鉴定意见书相关材料 → 审核 → 出具鉴定意见书 → 用章 → 归档 → 工作查询查看详情。

如果当前系统流程配置与上述名称略有差异，请以手册和现有流程定义为准，但必须覆盖“从收案到办结归档再到查询追溯”的完整闭环。

请完成以下工作：

1. 准备验收环境
   - 确认前后端可访问。
   - 确认数据库连接正常。
   - 确认用于试跑的业务账号可登录。
   - 从导出 Excel 或其 CSV/JSON 预览中选择主干流程测试案件。
   - 如果缺少必要数据，请创建最小可用测试案件。

2. 使用真实业务账号登录
   - 不允许只使用 admin 验收。
   - 至少使用主干流程涉及的业务角色账号。
   - 记录每个账号进入系统后的菜单、待办、权限情况。

3. 发起或触发主干流程
   - 从真实页面发起“收到委托书”或对应入口。
   - 填写手册要求的关键字段。
   - 优先使用导出数据中的案件字段、流程名称、状态和历史记录作为填报或对照依据。
   - 上传必要附件。
   - 保存并提交。
   - 记录流程实例编号、案件编号或可追溯标识。

4. 按角色流转办理
   - 使用不同角色账号接力办理。
   - 每个节点都要从待办进入办理页面。
   - 每个节点都要验证表单字段、附件、办理意见、提交按钮、下一步处理人。
   - 涉及审核、审批、分派、退回、重新提交的节点必须实际操作。
   - 至少执行一次退回，并验证退回后可重新提交。
   - 验证流程日志是否正确记录每一步。

5. 验证附件能力
   - 上传附件。
   - 下载附件。
   - 预览附件。
   - 验证附件是否在后续节点、办结详情、工作查询中可见。
   - 验证权限外账号是否不能越权访问附件。

6. 验证办结和只读
   - 将主干流程完整办结。
   - 办结后确认状态正确。
   - 办结后确认详情页为只读，不允许继续编辑关键业务字段。
   - 确认办结数据没有丢失。

7. 验证工作查询和追溯
   - 在工作查询中按案件编号、流程名称、状态、时间等条件查询。
   - 打开详情页。
   - 验证表单、附件、办理意见、流程日志、归档信息是否完整。
   - 对照导出 Excel、CSV/JSON 预览和原 OA 样本，检查流程名称、字段、状态、查询习惯是否存在关键偏差。

8. 验证权限边界
   - 使用无关角色账号尝试查看或办理该流程。
   - 验证菜单不可见、数据不可见或操作不可用是否符合要求。
   - 记录是否存在越权访问、越权办理、越权下载附件。

9. 不通过项处理规则
   - 主干流程中任何手册范围内失败项，必须当场修复。
   - 修复后必须重新跑相关节点或相关功能。
   - 不允许把主干流程关键失败项标记为“后续优化”。

10. 更新验收文档
   - 更新 docs/manual-scope-acceptance-template.md。
   - 更新 docs/judicial-appraisal-master-checklist.md。
   - 更新 docs/judicial-appraisal-flow-verification-matrix.md。
   - 新增或更新浏览器验收记录。
   - 更新 checklist.md。
   - 更新 GEMINI.md。

输出结果：
请在本次操作结束时输出以下内容：

1. 本次试跑的主干流程名称。
2. 本次使用的导出数据：
   - Excel 文件路径。
   - CSV/JSON/metadata 对照文件路径。
   - 选取的案件或样本记录。
   - 哪些字段用于测试或比对。
3. 本次使用的账号和角色：
   - 账号。
   - 角色。
   - 执行了哪些节点。
4. 本次创建或使用的案件/流程实例：
   - 案件编号。
   - 流程实例编号。
   - 发起时间。
   - 办结时间。
5. 主干流程节点流转结果表：
   - 节点名称。
   - 办理账号。
   - 页面是否可进入。
   - 表单是否正确。
   - 附件是否正确。
   - 意见是否保存。
   - 日志是否正确。
   - 是否通过。
6. 退回与重新提交验证结果。
7. 附件上传、下载、预览、权限验证结果。
8. 办结只读验证结果。
9. 工作查询和详情追溯验证结果。
10. 权限边界验证结果。
11. 本次发现的问题清单：
    - 问题编号。
    - 问题描述。
    - 影响范围。
    - 是否手册范围。
    - 修复方式。
    - 复验结果。
12. 本次修改了哪些代码或配置文件。
13. 本次更新了哪些验收文档。
14. 主干流程最终状态：
    - [x] 通过。
    - [~] 部分通过，说明剩余卡点。
    - [ ] 未通过，说明阻塞原因。
15. 是否建议进入“19 个流程逐项验收”：
    - 可以 / 不可以。
    - 原因。
```
