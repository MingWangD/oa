# 提示词 5：最终结项审计

```text
请执行司法鉴定管理系统“最终结项审计”。

目标：
在 19 个流程验收和公共功能专项验收之后，进行一次反向审计和结项判断。不要只看 checklist 是否打勾，而要从手册、流程、角色、数据、证据、权限、原 OA 对照多个方向反查系统是否真正达到可结项状态。

验收依据：
1. docs/司法鉴定系统使用手册.docx
2. docs/manual-scope-acceptance-template.md
3. docs/judicial-appraisal-master-checklist.md
4. docs/judicial-appraisal-flow-verification-matrix.md
5. docs/judicial-appraisal-manual-based-requirements-v2.md
6. docs/judicial-appraisal-flowchart-baseline.md
7. 全部 browser validation record
8. docs/original-oa-samples/
9. checklist.md
10. GEMINI.md
11. AGENT.md
12. 当前代码、配置、数据库、前后端运行结果

测试数据要求：
项目中之前已经导出过 Excel 表格数据，可以用于测试。最终结项审计必须反查 docs/original-oa-samples/exports/ 下的 Excel 导出文件，并结合 docs/original-oa-samples/csv-preview/、docs/original-oa-samples/json/、docs/original-oa-samples/metadata/ 和 docs/original-oa-samples/raw/，确认本系统的流程名称、字段、状态、查询结果、报表口径、归档习惯和权限数据范围与原 OA 样本不存在关键业务偏差。不要只凭新系统内部数据判断结项。

请完成以下审计：

1. 从手册目录反查系统
   - 阅读手册范围。
   - 按手册章节逐项反查系统是否有对应页面、流程、字段、角色、附件、查询、归档能力。
   - 如果手册有要求但系统无实现，必须列为结项阻塞。
   - 如果系统实现与手册不一致，判断是否属于关键偏差。

2. 从 19 个流程反查证据
   - 检查每个流程是否已完成真实页面验收。
   - 检查每个流程是否有真实业务账号。
   - 检查每个流程是否有案件编号或流程实例编号。
   - 检查每个流程是否跑到办结或手册定义的终态。
   - 检查每个流程是否验证附件、意见、日志、退回、工作查询、权限。
   - 检查每个流程是否使用或对照了导出 Excel、CSV/JSON 和 metadata。
   - 没有证据的 [x] 必须降级为 [~] 或 [ ]。

3. 从公共功能反查证据
   - 登录、菜单权限、待办、已办、我的发起、工作查询、流程详情。
   - 附件、办理意见、流程日志、退回、办结只读。
   - 归档、知识库、报表统计、数据权限。
   - 所有公共能力必须有运行证据，不能只凭代码存在判断通过。

4. 从角色权限反查风险
   - 使用不同业务角色审计菜单、流程入口、待办、查询、附件、详情。
   - 检查是否存在越权查看、越权办理、越权下载附件。
   - 检查是否存在该看却看不到、该办却办不了的问题。
   - 直接调用接口也要抽查后端权限。

5. 从工作查询和办结详情反查完整性
   - 抽查已办结流程。
   - 检查工作查询是否能查到。
   - 检查详情页是否完整。
   - 检查附件、意见、日志、状态是否完整。
   - 检查办结后是否只读。
   - 检查查询字段是否与手册、导出 Excel 和原 OA 样本一致。

6. 从原 OA 样本反查等价性
   - 对照 docs/original-oa-samples/exports/、csv-preview、json、metadata、raw。
   - 检查流程名称、流程分类、查询字段、状态习惯、导出或报表数据口径。
   - 判断是否存在影响业务迁移或业务理解的关键差异。

7. 从文档状态反查一致性
   - 检查以下文档状态是否一致：
     docs/manual-scope-acceptance-template.md
     docs/judicial-appraisal-master-checklist.md
     docs/judicial-appraisal-flow-verification-matrix.md
     browser validation record
     checklist.md
     GEMINI.md
   - 如果同一事项在不同文档状态不一致，必须修正。
   - 如果没有证据支撑 [x]，必须降级。
   - 如果已修复并复验通过，必须同步升级并写明证据。

8. 从代码和运行结果反查
   - 确认当前代码可以构建或测试通过。
   - 运行可用的后端测试。
   - 运行可用的前端构建或测试。
   - 如果测试无法运行，说明原因。
   - 如果测试失败且属于手册范围，必须修复并复验。
   - 如果只是文档审计，也要说明未运行代码测试的原因。

9. 结项判断标准
   只有同时满足以下条件，才允许建议结项：
   - 19 个手册范围流程全部为 [x]。
   - 所有公共功能全部为 [x]。
   - 所有 [~] 和 [ ] 已清零，或明确证明不属于当前手册范围。
   - 每个 [x] 都有真实页面、真实账号、真实流程实例证据。
   - 每个关键流程都已使用或对照导出 Excel、CSV/JSON 和 metadata。
   - 所有手册范围内验收失败项都已修复并复验通过。
   - 非 admin 业务账号能完成各自职责。
   - 工作查询、办结详情、附件、日志、意见、归档均可追溯。
   - 权限边界通过验证，不存在明显越权。
   - 原 OA 样本对照没有关键业务偏差。
   - 所有验收文档状态一致。

10. 不通过项处理规则
   - 如果发现结项阻塞问题，必须优先修复并复验。
   - 如果本次无法修复，必须明确列为结项阻塞，不允许建议结项。
   - 不允许把手册范围内的失败项写成“后续优化”。

输出结果：
请在本次操作结束时输出以下内容：

1. 最终结项审计结论：
   - 可以结项 / 暂不能结项。
   - 一句话说明核心原因。
2. 本次审计使用的导出数据：
   - Excel 文件路径。
   - CSV/JSON/metadata/raw 对照文件路径。
   - 覆盖了哪些流程、字段、状态和查询口径。
   - 是否存在导出数据无法覆盖的场景。
3. 19 个流程最终状态表：
   - 流程名称。
   - 状态：[x] / [~] / [ ]。
   - 证据是否充分。
   - 是否存在阻塞问题。
   - 备注。
4. 公共功能最终状态表：
   - 功能名称。
   - 状态：[x] / [~] / [ ]。
   - 证据是否充分。
   - 是否存在阻塞问题。
   - 备注。
5. 角色权限最终审计表：
   - 角色。
   - 菜单权限。
   - 流程权限。
   - 查询权限。
   - 附件权限。
   - 是否存在越权。
   - 是否通过。
6. 工作查询和办结追溯审计结果：
   - 抽查流程。
   - 案件编号或流程实例编号。
   - 是否可查。
   - 详情是否完整。
   - 附件是否完整。
   - 意见是否完整。
   - 日志是否完整。
   - 是否只读。
7. 原 OA 样本对照结果：
   - 对照范围。
   - 是否存在关键偏差。
   - 偏差说明。
   - 是否影响结项。
8. 文档一致性检查结果：
   - manual-scope-acceptance-template.md。
   - judicial-appraisal-master-checklist.md。
   - judicial-appraisal-flow-verification-matrix.md。
   - browser validation record。
   - checklist.md。
   - GEMINI.md。
   - 是否一致。
   - 修正了哪些不一致。
9. 测试和构建结果：
   - 后端测试命令。
   - 后端测试结果。
   - 前端测试或构建命令。
   - 前端测试或构建结果。
   - 未运行项及原因。
10. 本次发现的结项阻塞问题：
    - 问题编号。
    - 问题描述。
    - 所属流程或功能。
    - 是否手册范围。
    - 当前状态。
    - 修复建议。
    - 是否必须结项前完成。
11. 本次已修复并复验的问题：
    - 问题编号。
    - 修改文件。
    - 修复内容。
    - 复验方式。
    - 复验结果。
12. 最终剩余清单：
    - 必须结项前完成。
    - 可结项后优化。
    - 不属于当前手册范围。
13. 最终建议：
    - 是否建议结项。
    - 如果建议结项，请说明满足了哪些结项条件。
    - 如果不建议结项，请给出下一轮最小修复计划。
```
