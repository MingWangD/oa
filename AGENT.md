# Codex 协作约定

## 用户使用习惯

- 每次开始任务前，必须先阅读项目根目录下的 `GEMINI.md`，以获取最新的开发进度、Gemini CLI 的工作记录以及当前的上下文对齐信息。
- 直接在当前 `oa` 项目中工作，并基于现有实现持续开发。
- 保留并扩展已有代码；没有明确技术原因时，不要重写或废弃现有结构。
- 每次开始任务前，先查找是否存在适用的 Codex 技能；如果没有对应技能，再按照常规工程方式处理。
- 已克隆的 Anthropic Skills 参考仓库保存在 `vendor/anthropic-skills`，供后续检查、参考和适配。
- 如果本地缺少任务需要的工具或环境，应帮助安装，并在安装完成后继续验证。
- 进行较大代码修改前，先阅读最新需求文档并检查当前代码状态。
- 每次修改代码后，运行可用的前后端测试和构建。如果环境阻碍验证，应明确说明具体原因。
- 不得覆盖或删除用户已有修改。遇到来源不明的本地改动时，默认视为用户修改，除非用户另有说明。
- 用户明确要求“无需每一步都汇报”时，应减少过程性汇报，尽量连续完成整批任务后再统一汇总。
- 进入阶段性收官或清单式任务时，应优先采用“尽量多做、一次性交付”的推进方式，而不是每完成一小步就停下来确认。

## 需求优先级

需求来源发生冲突时，严格按照以下顺序处理：

1. 《完整OA系统重构需求规格说明书.docx》
2. 《司法鉴定系统使用手册.docx》及细化流程图
3. 现有 OA 网站的实际功能和行为
4. 当前项目中的现有代码
5. `docs/` 目录下的旧 Markdown 文档

部分旧 Markdown 文档仅描述司法鉴定一期简化方案。不得使用这些旧文档缩小当前项目范围，当前目标是重构完整 OA 系统。

## 产品范围

目标是完整复刻管理员账号可见的全部 OA 模块，包括：

- 快捷菜单、个人事务、流程中心、应用中心
- CRM、绩效、合同、项目、仓库、安全风险
- 行政办公、知识管理、督查督办、智能门户、报表中心
- 人力资源、考勤、公文、档案、交流园地
- 开放与集成平台、附件程序、系统管理
- 司法鉴定完整业务

界面允许进行现代化重构，但业务功能、数据含义、流程结果、权限边界和审计结果必须保持等价。

## 技术方向

- 继续使用 Spring Boot 3、Java 17、Spring Security、MyBatis-Plus、MySQL、Redis、MinIO。
- 继续使用 Vue 3、TypeScript、Vite、Pinia、Vue Router、Element Plus。
- 优先建设公共平台能力，包括菜单、RBAC、数据权限、动态表单、动态流程、文件、审计、知识归档、报表和数据源。
- 公共平台能力完成后，再逐步实现各业务模块。

## 司法鉴定必须实现的能力

- 20 个流程定义和 19 个表单定义。
- 输入文件、输出文件及注意事项/业务约束模型。
- 逐节点办理角色、条件分支、退回路径和归档要求。
- 并行任务、子流程、流程关联、异议、出庭、退费、终止和归档。
- 完整保留鉴定意见书初稿、A、A-B、A-B-C、最终送审稿版本。
- 司法鉴定专属角色：部门负责人、项目负责人、项目辅助人、档案管理员、中心档案管理员、技术负责人、审阅所长、综合业务部、财务、收案员。
